> 수백 시간의 시행착오에서 뽑아낸, "그때 알았더라면" 싶은 팁들을 모았습니다.

## 프롬프트 작성 원칙 5가지

### 1) 구체적으로, 숫자로

!!! warning "⚠️ 나쁜 예"
    ```
    코드 좀 개선해줘
    ```

!!! tip "💡 좋은 예"
    ```
    src/services/payment.py 의 process_payment 함수를 리팩토링해줘.
    - 복잡도 순환 10 이하
    - 함수 크기 50줄 이하
    - 예외 처리 3가지: 잔액 부족, 카드 만료, 네트워크 오류
    - pytest 테스트 5개 추가
    ```

### 2) 제약 조건을 먼저

"할 일 → 하지 말 것" 순서로 쓰세요.

```text
다음을 수행해줘:
- [해야 할 일 1]
- [해야 할 일 2]

단, 다음은 건드리지 마:
- package.json의 버전
- 기존 테스트 파일
```

### 3) 출력 형식 명시

"표", "Mermaid 다이어그램", "YAML", "bullet points" 같이 형식을 지정하면 결과 품질이 크게 오릅니다.

### 4) 참조할 파일 명시

```text
@src/models/user.py 의 스타일을 참고해서
@src/models/order.py 의 DTO를 작성해줘.
```

### 5) 플랜 먼저, 실행은 나중

복잡한 작업은 2단계로:

```text
1단계: "먼저 계획만 세워서 보여줘. 실제 수정은 하지 마."
2단계: (계획 검토 후) "1단계 3번 항목만 실행해줘."
```

## GSTACK 스킬 사용 팁

### `/office-hours` — "당신의 가정을 부수는 질문"

- 아이디어가 떠오른 직후에 바로 실행하지 말고, 24시간 묵힌 후 실행하세요
- Claude의 비판적 질문에 방어하지 말고 솔직히 답하세요
- 답변을 노션/옵시디언에 기록해두고 6개월 후 다시 보세요

### `/review` — "PR마다 한 번"

- `git commit` 직후가 아니라 `gh pr create` **전**에 실행
- Critical 이슈는 반드시 해결, Warning은 선별적으로
- 반복되는 Warning 패턴은 ESLint/pylint 룰로 자동화

### `/think` — "감을 논리로"

- "그냥 A가 맞을 것 같아"라는 느낌이 들 때 실행
- 3개 이상의 대안을 제시하게 하세요 ("3개 대안을 비교")
- 결정 이유를 ADR(Architecture Decision Record)로 보관

### `/ship` — "금요일 오후 금지"

- 월-목 오전에만 실행
- 체크리스트 한 항목이라도 실패하면 배포 중단
- 배포 후 30분은 모니터링 화면 주시

!!! danger "🚨 절대 하지 말 것"
    - GSTACK이 제안한 코드를 **읽지 않고** 그대로 머지
    - `/ship`을 건너뛰고 `git push --force origin main`
    - 프로덕션 DB에 직접 MCP 연결 (읽기 전용 계정 쓰기)

## 토큰 비용 관리

Claude Code는 API 호출마다 토큰을 소모합니다. 대학생에게는 비용이 아깝죠.

!!! tip "💡 토큰 절약 10계명"
    1. 큰 파일은 필요한 부분만 `@file.py:10-50` 처럼 범위 지정
    2. "간결하게" / "핵심만" 같은 지시를 습관화
    3. 반복 작업은 GSTACK 스킬로 프롬프트 재사용
    4. 긴 문서 요약은 한 번만 하고 Memory MCP에 저장
    5. 디버깅 시 전체 스택 트레이스 대신 핵심 라인만 붙여넣기
    6. 불필요한 컨텍스트(`tree` 출력 전체 등) 포함하지 않기
    7. Plan 모드로 헛수고 방지
    8. 월별 토큰 사용량 대시보드 체크
    9. 토큰 비싼 작업(대규모 분석)은 오프피크에
    10. 무료 티어로도 충분한 작업은 굳이 유료 API 쓰지 않기

## 혼자 개발할 때의 마인드셋

대학생은 보통 팀 없이 혼자 개발합니다. GSTACK + MCP는 **AI 팀원** 역할을 대신해줄 수 있습니다.

```text
[AI 팀원 역할 분담]
- CEO (전략)    → /office-hours
- CTO (아키텍처) → /plan-eng-review
- QA (품질)     → /qa
- 보안 (CSO)    → /cso
- 배포 (DevOps) → /ship, /canary
```

!!! info "📌 핵심 마인드셋"
    "혼자 개발하지만 팀처럼 체크리스트를 지킨다"가 핵심입니다. 코드는 혼자 쓰더라도, **검토 기준은 팀 수준**으로 유지하세요.

## 흔한 실수와 해결

### 실수 1: 프롬프트가 모호함
- **증상:** Claude가 엉뚱한 파일을 수정
- **해결:** 파일 경로를 명시적으로 `@path/to/file`로 지정

### 실수 2: Plan 모드를 안 씀
- **증상:** 10개 파일이 망가짐
- **해결:** 파일 2개 이상 수정은 기본 Plan 모드

### 실수 3: MCP 권한 과다
- **증상:** 실수로 프로덕션 DB에 DELETE
- **해결:** 읽기 전용 계정, 최소 권한

### 실수 4: 토큰 폭탄
- **증상:** 한 달 API 비용 10만 원
- **해결:** 범위 지정, 요약 재활용, Memory MCP

### 실수 5: Claude 답변 맹신
- **증상:** 존재하지 않는 함수 사용
- **해결:** 생성된 코드를 꼭 한 번 읽고, 테스트 실행

!!! danger "🚨 AI 환각(Hallucination) 경계"
    Claude도 가끔 "있을 법한" 함수나 라이브러리를 만들어냅니다. 공식 문서로 교차 검증하는 습관을 들이세요.

## 학습 로드맵 (4주 완성)

**Week 1 — 기초**
- Claude Code 설치, `Shift+Tab` 몸으로 익히기
- Filesystem MCP, GitHub MCP 2개만 설치
- 매일 `/review`, `/daily` 사용

**Week 2 — 확장**
- Postgres MCP, Playwright MCP 추가
- Sequential Thinking 프롬프트 시도
- 첫 Plan 모드 대규모 리팩토링 경험

**Week 3 — GSTACK**
- GSTACK 설치, 23개 스킬 훑기
- `/office-hours`, `/plan-eng-review`, `/ship` 각 1회
- 자기 프로젝트에 매일 루틴 적용

**Week 4 — 통합**
- MCP + GSTACK 조합 프롬프트 설계
- 자기만의 프롬프트 템플릿 5개 작성
- 학습한 것을 블로그/노션에 정리

!!! tip "💡 꿀팁: 매주 금요일 `/retro`"
    매주 금요일에 `/retro`를 실행하고 "이번 주에 가장 도움이 된 스킬 TOP 3"를 기록하세요. 3개월 후 당신만의 워크플로우가 완성됩니다.
